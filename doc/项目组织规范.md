# TextRPG 项目组织规范

本文档定义 TextRPG 项目的组织结构、层次划分和编码规范。所有代码变更应遵循本文档规定。

---

## 1. 核心原则

### 1.1 层次分离

项目采用 **三层架构**：

| 层次 | 职责 | 依赖方向 |
|------|------|----------|
| **domain** | 核心业务模型、领域实体、领域枚举、领域服务接口 | 被依赖，不依赖其他层 |
| **data** | 数据持久化、配置管理、仓储实现、外部数据源适配 | 依赖 domain |
| **adapter** | 外部服务对接（LLM、消息平台、文件系统等） | 依赖 domain |
| **utils** | 通用工具类，不涉及业务逻辑 | 被广泛依赖，不限制 |
| **application** | 应用编排、入口配置 | 协调各层 |

### 1.2 依赖规则

```
domain          ← 无任何外部依赖
  ↑
data / adapter  ← 仅依赖 domain
  ↑
application     ← 依赖 domain、data、adapter
```

- **domain 层禁止依赖 data、adapter、application 层**
- **data 层禁止依赖 adapter 层**
- 同层之间可互相依赖

### 1.3 单一职责

每个模块（文件）应有明确的单一职责。当模块过大或职责模糊时，应拆分为多个模块。

### 2.2 模块扩展原则

当需要新增功能模块时：

1. **确定所属层次**：根据职责判断属于哪个层次
2. **创建对应目录**：在对应层次下创建 `*/` 目录
3. **遵循命名规范**：目录名和文件名使用 PascalCase
4. **同步更新本文档**：如有新的通用模式，需补充本文档

## 4. 领域层规范

### 4.1 领域模型 (`domain/model/`)

存放领域枚举和不可变值对象。

**枚举类规范**：
- 每个枚举需有 `fromValue(value: T)` 伴生对象方法
- 使用 `entries` 而非 `values()` 遍历

```kotlin
enum class ItemType(val value: Int) {
    EQUIPMENT(1),
    CONSUMABLE(2),
    MATERIAL(3),
    QUEST(4);

    companion object {
        fun fromValue(value: Int): ItemType = entries.first { it.value == value }
    }
}
```

**值对象规范**：
- 使用 `value class` 或不可变 `data class`
- 无唯一标识，仅通过属性值相等

### 4.2 领域实体 (`domain/entity/`)

存放领域实体类，**不包含任何持久化相关注解**。

**实体类规范**：
- 纯 Kotlin 类，无框架注解
- 不直接依赖数据库框架
- 包含业务方法和验证逻辑

```kotlin
data class Player(
    val id: Long = 0,
    val name: String,
    val bindAccount: String,
    val createdAt: DateTime? = null,
    val updatedAt: DateTime? = null
) {
    fun rename(newName: String): Player = copy(name = newName, updatedAt = DateTime.now())
}
```

### 4.3 领域服务接口 (`domain/service/`)

定义领域服务的接口，不包含实现。

**服务接口规范**：
- 以 `I` 为前缀（可选）
- 方法参数和返回值使用领域模型
- 不依赖具体技术栈

```kotlin
interface IPlayerService {
    fun findById(id: Long): Player?
    fun findByBindAccount(account: String): Player?
    fun create(name: String, bindAccount: String): Player
}
```

## 5. 数据层规范

### 5.1 数据库表定义 (`data/database/`)

使用 Exposed ORM 定义数据库表结构。

**表定义规范**：
- 继承 `LongIdTable` 或 `IntIdTable`
- 表名使用复数形式
- 列名使用 snake_case

```kotlin
object Players : LongIdTable("players") {
    val name = varchar("name", 64).uniqueIndex()
    val bindAccount = varchar("bind_account", 128)
    val createdAt = datetime("created_at")
    val updatedAt = datetime("updated_at")
}
```

### 5.2 数据库工厂 (`data/database/DatabaseFactory.kt`)

负责数据库连接管理和初始化。

```kotlin
class DatabaseFactory(private val config: DatabaseConfig) {
    val database: Database by lazy {
        Database.connect(url = config.url, driver = config.driver)
    }

    fun init() {
        transaction(database) {
            SchemaUtils.createMissingTablesAndColumns(...)
        }
    }
}
```

### 5.3 仓储模式

#### 5.3.1 仓储接口 (`IRepository.kt`)

```kotlin
interface IRepository<T, ID> {
    fun findById(id: ID): T?
    fun findAll(): List<T>
    fun save(entity: T): T
    fun delete(id: ID): Boolean
}
```

#### 5.3.2 仓储实现

- 构造参数仅接受 `Database` 实例
- 数据库操作包裹在 `transaction(database)` 中
- 私有转换方法：`{Entity}Entity.to{Model}()`

```kotlin
class PlayerRepository(private val database: Database) : IRepository<Player, Long> {
    override fun findById(id: Long): Player? = transaction(database) {
        PlayerEntity.findById(id)?.toPlayer()
    }

    private fun PlayerEntity.toPlayer() = Player(
        id = id.value,
        name = name,
        bindAccount = bindAccount,
        createdAt = createdAt,
        updatedAt = updatedAt
    )
}
```

### 5.4 数据模型映射

| 数据库层 | 应用层 | 说明 |
|----------|--------|------|
| `Table` | 无直接对应 | 表定义 |
| `*Entity` | 无直接对应 | Exposed DAO |
| 无 | `{Entity}` | 领域模型（data class） |

数据流转：`Table` ↔ `*Entity` ↔ `{Entity}``

## 7. 依赖注入规范

### 7.1 Koin 模块组织

| 文件 | 导出 | 依赖 |
|------|------|------|
| `data/DatabaseModules.kt` | `databaseModule`、`repositoryModule` | AppConfig |

### 7.2 模块注册顺序

```kotlin
startKoin {
    modules(appModules, databaseModule, repositoryModule)
}
```

**顺序规则**：
1. `databaseModule` 必须在使用 `Database` 的模块之前
2. `repositoryModule` 必须在需要仓储的模块之前
3. `appModules` 最后注册（汇总）

### 7.3 模块定义模板

```kotlin
// data/DatabaseModules.kt
val databaseModule = module {
    single { DatabaseFactory(get<AppConfig>().database).also { it.init() } }
    single { get<DatabaseFactory>().database }
}

val repositoryModule = module {
    singleOf(::PlayerRepository)
    singleOf(::ItemRepository)
}
```

## 8. 编码规范

### 8.1 类注释

使用 KDoc 风格：

```kotlin
/**
 * 类的简要描述
 *
 * 类的详细说明（可选）
 *
 * @property prop1 属性1说明
 * @property prop2 属性2说明
 */
class ExampleClass { ... }
```

### 8.2 方法注释

```kotlin
/**
 * 方法的简要描述
 *
 * @param param1 参数1说明
 * @param param2 参数2说明
 * @return 返回值说明
 * @throws ExceptionName 何时抛出此异常
 */
fun method(param1: Type1, param2: Type2): ReturnType { ... }
```

### 8.3 错误处理

- 使用 `Result<T>` 作为可能失败操作的返回类型
- 使用 `error("message")` 抛出明确的业务异常
- 避免裸露的 `try-catch`，优先使用 `runCatching`

```kotlin
fun findById(id: Long): Player? = transaction(database) {
    PlayerEntity.findById(id)?.toPlayer()
}

fun update(entity: Player): Player = transaction(database) {
    val existing = PlayerEntity.findById(entity.id)
        ?: error("Player not found: ${entity.id}")
    // ...
}
```

3. 检查并更新所有 import 语句
4. 验证编译通过

## 14. 常见模式

### 13.1 工厂模式

当构造逻辑复杂时，使用工厂类：

```kotlin
class PlayerFactory {
    fun create(name: String, account: String): Player {
        return Player(
            name = name,
            bindAccount = account,
            createdAt = DateTime.now(),
            updatedAt = DateTime.now()
        )
    }
}
```

### 13.2 策略模式

当行为有多种变体时：

```kotlin
interface DamageCalculator {
    fun calculate(baseDamage: Int, target: Player): Int
}

class FireDamageCalculator : DamageCalculator { ... }
class IceDamageCalculator : DamageCalculator { ... }
```

### 13.3 观察者模式

当需要解耦事件发送和接收时：

```kotlin
interface GameEventListener {
    fun onEvent(event: GameEvent)
}

class GameEventBus {
    private val listeners = mutableListOf<GameEventListener>()
    fun subscribe(listener: GameEventListener) { ... }
    fun unsubscribe(listener: GameEventListener) { ... }
    fun publish(event: GameEvent) { listeners.forEach { it.onEvent(event) } }
}
```
