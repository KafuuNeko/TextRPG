# TextRPG 项目组织规范

本文档定义 TextRPG 项目的组织结构、层次划分和编码规范。所有代码变更应遵循本文档规定。

---

## 1. 核心原则

### 1.1 层次分离

项目采用 **三层架构**：

| 层次 | 职责 | 依赖方向 |
|------|------|----------|
| **domain** | 核心业务模型、领域实体、领域枚举、领域服务接口 | 被依赖，不依赖其他层 |
| **data** | 数据持久化、配置管理、仓储实现、外部数据源适配 | 依赖 domain |
| **adapter** | 外部服务对接（LLM、消息平台、文件系统等） | 依赖 domain |
| **utils** | 通用工具类，不涉及业务逻辑 | 被广泛依赖，不限制 |
| **application** | 应用编排、入口配置 | 协调各层 |

### 1.2 依赖规则

```
domain          ← 无任何外部依赖
  ↑
data / adapter  ← 仅依赖 domain
  ↑
application     ← 依赖 domain、data、adapter
```

- **domain 层禁止依赖 data、adapter、application 层**
- **data 层禁止依赖 adapter 层**
- 同层之间可互相依赖

### 1.3 单一职责

每个模块（文件）应有明确的单一职责。当模块过大或职责模糊时，应拆分为多个模块。

### 2.2 模块扩展原则

当需要新增功能模块时：

1. **确定所属层次**：根据职责判断属于哪个层次
2. **创建对应目录**：在对应层次下创建 `*/` 目录
3. **遵循命名规范**：目录名和文件名使用 PascalCase
4. **同步更新本文档**：如有新的通用模式，需补充本文档

## 4. 领域层规范

### 4.1 领域模型 (`domain/model/`)

存放领域枚举和不可变值对象。

**枚举类规范**：
- 每个枚举需有 `fromValue(value: T)` 伴生对象方法
- 使用 `entries` 而非 `values()` 遍历

```kotlin
enum class ItemType(val value: Int) {
    EQUIPMENT(1),
    CONSUMABLE(2),
    MATERIAL(3),
    QUEST(4);

    companion object {
        fun fromValue(value: Int): ItemType = entries.first { it.value == value }
    }
}
```

**值对象规范**：
- 使用 `value class` 或不可变 `data class`
- 无唯一标识，仅通过属性值相等

### 4.2 领域实体 (`domain/entity/`)

存放领域实体类，**不包含任何持久化相关注解**。

**实体类规范**：
- 纯 Kotlin 类，无框架注解
- 不直接依赖数据库框架
- 包含业务方法和验证逻辑

```kotlin
data class Player(
    val id: Long = 0,
    val name: String,
    val bindAccount: String,
    val createdAt: DateTime? = null,
    val updatedAt: DateTime? = null
) {
    fun rename(newName: String): Player = copy(name = newName, updatedAt = DateTime.now())
}
```

### 4.3 领域服务接口 (`domain/service/`)

定义领域服务的接口，不包含实现。

**服务接口规范**：
- 以 `I` 为前缀（可选）
- 方法参数和返回值使用领域模型
- 不依赖具体技术栈

```kotlin
interface IPlayerService {
    fun findById(id: Long): Player?
    fun findByBindAccount(account: String): Player?
    fun create(name: String, bindAccount: String): Player
}
```

## 5. 数据层规范

### 5.1 数据库表定义 (`data/database/`)

使用 Exposed ORM 定义数据库表结构。

**表定义规范**：
- 继承 `LongIdTable` 或 `IntIdTable`
- 表名使用复数形式
- 列名使用 snake_case

```kotlin
object Players : LongIdTable("players") {
    val name = varchar("name", 64).uniqueIndex()
    val bindAccount = varchar("bind_account", 128)
    val createdAt = datetime("created_at")
    val updatedAt = datetime("updated_at")
}
```

### 5.2 数据库工厂 (`data/database/DatabaseFactory.kt`)

负责数据库连接管理和初始化。

```kotlin
class DatabaseFactory(private val config: DatabaseConfig) {
    val database: Database by lazy {
        Database.connect(url = config.url, driver = config.driver)
    }

    fun init() {
        transaction(database) {
            SchemaUtils.createMissingTablesAndColumns(...)
        }
    }
}
```

### 5.3 仓储模式

#### 5.3.1 仓储接口 (`IRepository.kt`)

```kotlin
interface IRepository<T, ID> {
    fun findById(id: ID): T?
    fun findAll(): List<T>
    fun save(entity: T): T
    fun delete(id: ID): Boolean
}
```

#### 5.3.2 仓储实现

- 构造参数仅接受 `Database` 实例
- 数据库操作包裹在 `transaction(database)` 中
- 私有转换方法：`{Entity}Entity.to{Model}()`

```kotlin
class PlayerRepository(private val database: Database) : IRepository<Player, Long> {
    override fun findById(id: Long): Player? = transaction(database) {
        PlayerEntity.findById(id)?.toPlayer()
    }

    private fun PlayerEntity.toPlayer() = Player(
        id = id.value,
        name = name,
        bindAccount = bindAccount,
        createdAt = createdAt,
        updatedAt = updatedAt
    )
}
```

### 5.4 数据模型映射

| 所在层 / 目录 | 类型 | 说明 |
|-------------|------|------|
| `data/database/` | `object Table : LongIdTable` | Exposed 表定义 |
| `data/dao/` | `class *Entity : LongEntity` | Exposed DAO（类名保留 `*Entity` 后缀作为 DAO 惯例，文件名为 `*Dao.kt` 便于目录导航） |
| `domain/entity/` | `data class {Model}` | 纯 Kotlin 领域模型（遵循 §4.2：无框架依赖） |

数据流转方向：
```
DB ─ read ─▶ Table ⇄ *Entity ─ toXxx() ─▶ {Model}  ─ 返回给业务层
DB ◀ write ─ Table ⇄ *Entity ◀─ mutate ─ {Model}  ─ 由 Repository 接收
```

`Repository.save()` 等方法接收领域模型 `{Model}`，内部在 `transaction` 里转成/更新 `*Entity`，返回时再映射回 `{Model}`。业务层（`game/*`）只认领域模型，不接触 DAO。

## 7. 依赖注入规范

### 7.1 Koin 模块组织

框架层共交付 6 个 Koin 模块 + Bootstrap 单例；范例层实现自己的 `handlerModule` 和可选的覆盖模块。

| 文件 | 导出模块 | 职责 | 对外依赖 |
|------|---------|------|----------|
| `Application.kt`（私有 `appModule`） | `appModule`（私有） | HttpClient / AppConfig / OneBotAdapter / LLMClient / KotlinScriptRunner / CoroutineScope(gameScope) / ApplicationBootstrap | - |
| `data/ConfigModules.kt` | `configModule` | 10 个独立 ConfigLoader 产物注册 + `ConfigQualifiers` | appModule（需要 AppConfig 的 Loader 在 appModule） |
| `data/DatabaseModules.kt` | `databaseModule`、`repositoryModule` | DatabaseFactory / Database / PlayerRepository / ItemRepository / BuffRepository | appModule (AppConfig) |
| `game/GameModules.kt` | `gameModule` | 13 个 Manager/Engine/Service（MapManager、NpcManager、QuestManager、EffectEngine、SkillEngine、CombatSessionFactory 等） | configModule、appModule、repositoryModule |
| `game/ai/AiModule.kt` | `aiModule` | FunctionCallingClient / AIToolRegistry / PlayerTagManager / AISceneManager | appModule、configModule、gameModule |
| 范例层（如 `simple/.../HandlerModules.kt`） | `handlerModule`（范例） | 具体指令 Handler 类（`NamedCommandHandler` 实现），按 `singleOf(::XxxCommandHandler) bind NamedCommandHandler::class` 注册 | gameModule、aiModule 等 |

### 7.2 模块注册顺序

```kotlin
// Application.kt
startKoin {
    modules(
        appModule, configModule, databaseModule, repositoryModule,
        gameModule, aiModule
        // 范例层的 handlerModule 与覆盖模块列在此之后
    )
}
```

**顺序规则**（Koin 允许任意顺序，但按如下顺序列出便于错误时定位）：
1. `appModule` 第一（基础设施 + CoroutineScope）
2. `configModule` 在 `gameModule` 之前（配置被 Manager 消费）
3. `databaseModule` 在 `repositoryModule` 之前
4. `gameModule` 在 `aiModule` 之前（`BuiltinAITools.registerAll` 需要 3 个 Manager）
5. 范例层的 `handlerModule` 与覆盖模块最后列出

### 7.3 模块定义模板

**框架层 `configModule`（摘）**：
```kotlin
// data/ConfigModules.kt
object ConfigQualifiers {
    val AttributeDefinitions = named("attributeDefinitions")
    val BuffDefinitions      = named("buffDefinitions")
    val EnemyDefinitions     = named("enemyDefinitions")
    val QuestDefinitions     = named("questDefinitions")
}

val configModule = module {
    // 类型唯一的 data class 配置直接注册
    single<AIConfig> { AIConfigLoader.load() }
    single<MapConfig> { MapConfigLoader.load() }
    // ...

    // CombatConfigResult 拆分成全局战斗配置 + 敌人 Map
    single<CombatConfigResult> { CombatConfigLoader.load() }
    single<CombatConfig> { get<CombatConfigResult>().combatConfig }
    single<Map<String, EnemyDefinition>>(ConfigQualifiers.EnemyDefinitions) {
        get<CombatConfigResult>().enemies
    }
    single<Map<String, AttributeDefinition>>(ConfigQualifiers.AttributeDefinitions) {
        AttributeConfigLoader.load()
    }
    // ...
}
```

> **泛型类型擦除提醒**：Kotlin 的 `typealias` 在运行时被擦除为原始类型，**不能**用 typealias 区分多个 `Map<String, X>` 的 Koin 注册。必须用 `named()` 限定符。
>
> **Map qualifier 最小化**：仅当 Map 类型有真实下游消费方时才追加 qualifier。当前 4 个对应：`AttributeDefinitions` / `BuffDefinitions` / `EnemyDefinitions`（`CombatSessionFactory` 需要）+ `QuestDefinitions`（`QuestManager` 需要）。其他配置消费 data class 包装即可（如 `MapManager` 直接用 `MapConfig`）。

**范例层覆盖模板**：
```kotlin
val sampleOverrideModule = module {
    single<CooldownManager>(override = true) { /* 范例自定义 */ }
}
// startKoin 时列在框架模块之后
```

### 7.4 Bootstrap 启动编排

`ApplicationBootstrap` 定义了应用启动的三个钩子：
- `onBeforeStart()` — 启动前一次性初始化（典型：数据播种）
- `registerHandlers()` — 默认从 Koin 批量取 `NamedCommandHandler` 注册到 `CommandHandlerRegistry`
- `startLoop()` — 连接外部消息循环（默认 OneBot）

范例层扩展方式（任选其一）：
- 继承 `ApplicationBootstrap` 覆盖钩子，并在范例模块里 `single<ApplicationBootstrap>(override = true) { SampleBootstrap(...) }`
- 或在范例的 Bootstrap 子类构造器中直接调用 super 并追加逻辑

### 7.5 CommandHandler 注册规范

框架提供两种对称的 Handler 注册路径：

| 路径 | 接口 | 适用场景 |
|------|------|---------|
| 闭包式 | `fun interface CommandHandler` | 依赖 ≤ 2 / 逻辑 < 30 行 / 一次性实验 |
| 类 + Koin 批量 | `interface NamedCommandHandler : CommandHandler` | 依赖 ≥ 3 / 逻辑 ≥ 30 行 / 需要单测 / 希望被 Koin 构造器注入 |
| 脚本热更新 | `fun interface ScriptHandlerFactory` | `.kts` 动态编译 |

`NamedCommandHandler` **必须是普通 interface**，不能是 `fun interface`（因为含有 `name` 属性，SAM 转换不支持多成员接口）。

## 8. 编码规范

### 8.1 类注释

使用 KDoc 风格：

```kotlin
/**
 * 类的简要描述
 *
 * 类的详细说明（可选）
 *
 * @property prop1 属性1说明
 * @property prop2 属性2说明
 */
class ExampleClass { ... }
```

### 8.2 方法注释

```kotlin
/**
 * 方法的简要描述
 *
 * @param param1 参数1说明
 * @param param2 参数2说明
 * @return 返回值说明
 * @throws ExceptionName 何时抛出此异常
 */
fun method(param1: Type1, param2: Type2): ReturnType { ... }
```

### 8.3 错误处理

- 使用 `Result<T>` 作为可能失败操作的返回类型
- 使用 `error("message")` 抛出明确的业务异常
- 避免裸露的 `try-catch`，优先使用 `runCatching`

```kotlin
fun findById(id: Long): Player? = transaction(database) {
    PlayerEntity.findById(id)?.toPlayer()
}

fun update(entity: Player): Player = transaction(database) {
    val existing = PlayerEntity.findById(entity.id)
        ?: error("Player not found: ${entity.id}")
    // ...
}
```

## 10. 新增模块规范

### 10.1 当前实际结构：扁平

`game/` 下各子模块 (`combat/` / `quest/` / `npc/` / `map/` 等) 采用**扁平结构**——
直接把实体、Manager、Service、枚举等 `.kt` 文件放在子模块根目录下。实际组织见 §2.1 目录树。

```
game/combat/
├── CombatSession.kt
├── CombatEntity.kt
├── CombatSessionFactory.kt
├── CombatEffectContext.kt
└── EnemyAI.kt
```

### 10.2 新增子模块时的最小约束

新增 `game/xxx/` 子模块时，只需遵循：
1. **包名**：`org.textrpg.application.game.xxx`
2. **命名**：文件、类用 PascalCase；Koin 模块函数 camelCase
3. **DI 入口**：若需要进 Koin，统一追加到 [`game/GameModules.kt`](../src/main/kotlin/org/textrpg/application/game/GameModules.kt) 的 `gameModule`（不推荐每个子模块单独一个 Koin 模块——除非像 `ai/` 这样体量大的独立子系统）
4. **同步本规范**：更新 §2.1 目录树与 §7.1 Koin 模块表

> **历史说明**：本节早期版本曾规划 `model/entity/service/repository/` 四层嵌套结构，但实际代码沿用扁平结构演进。扁平结构在当前项目规模下更清晰，故已按现实对齐。若未来某个子模块体量增长到需要拆分（如 `game/combat/` 膨胀到 15+ 文件），再按需引入子目录。

工具类存放通用功能，与业务逻辑无关：

```
utils/
├── script/                 # 脚本运行时
│   └── KotlinScriptRunner.kt
├── string/               # 字符串处理（未来）
│   └── *.kt
└── math/                 # 数学计算（未来）
    └── *.kt
```

**规范**：
- 工具类应为纯函数或单例对象
- 不持有业务状态
- 不依赖 domain 层以外的任何模块

## 14. 常见模式

### 13.1 工厂模式

当构造逻辑复杂时，使用工厂类：

```kotlin
class PlayerFactory {
    fun create(name: String, account: String): Player {
        return Player(
            name = name,
            bindAccount = account,
            createdAt = DateTime.now(),
            updatedAt = DateTime.now()
        )
    }
}
```

### 13.2 策略模式

当行为有多种变体时：

```kotlin
interface DamageCalculator {
    fun calculate(baseDamage: Int, target: Player): Int
}

class FireDamageCalculator : DamageCalculator { ... }
class IceDamageCalculator : DamageCalculator { ... }
```

### 13.3 观察者模式

当需要解耦事件发送和接收时：

```kotlin
interface GameEventListener {
    fun onEvent(event: GameEvent)
}

class GameEventBus {
    private val listeners = mutableListOf<GameEventListener>()
    fun subscribe(listener: GameEventListener) { ... }
    fun unsubscribe(listener: GameEventListener) { ... }
    fun publish(event: GameEvent) { listeners.forEach { it.onEvent(event) } }
}
```
