## 〇、文档定位

### 0.1 为什么拆两份

原 `待修改.md` 基于 `simple` 分支的视角写成，把「Application.kt 膨胀」「11 个闭包 handler」这类范例工程问题与「DI 模式缺失」「AI 模块未接入」这类框架能力缺失混在一起。先生现在在 `develop` 分支工作——**对 develop 而言，Application.kt 本来就是 82 行，没有"拆"的问题，只有"补"的问题**。

本文档只谈框架层该补什么。simple 分支的工程重构见 [范例层待修改.md](范例层待修改.md)。

### 0.2 框架层 vs 范例层的判断标准

沿用 [开发计划.md](开发计划.md) §1 的定义：
- **框架层**：引擎、抽象机制、通用约定。被依赖，不依赖业务语义。
- **范例层**：具体游戏内容，基于框架能力拼装。

一个改动归框架层的充分条件（满足任意一条）：
1. 未来所有游戏范例都会遇到同样的问题
2. 解决它需要新增/修改共享的抽象（接口、注解、DI 约定、规范条文）
3. 不解决它，任何范例在接入该子系统时都必须重复造轮子

## 二、框架层缺失清单

### F1 · `config` 层缺少 Koin 模块

**现状**：
- `data/config/` 下有 **10 个独立 `XxxConfigLoader`**（AI / Attribute / Buff / Combat / Command / Inventory / Map / Npc / Quest / Skill）。注意：`EffectConfigLoader` 和 `RequiresParser` 是**共享解析工具**（提供 `parseEffectList` / `parseRequires` 给其他 Loader 复用），不是独立的配置加载器，本 F1 不涉及它们
- 所有独立 Loader 均为 `object`，接口形如 `load(path: String = DEFAULT_PATH): XxxConfig`
- 只有 `AppConfig` 进了 `appModule`（`Application.kt:35`），其余 10 个**未提供任何 Koin 模块**
- 任何使用这些配置的范例都必须手动 `XxxConfigLoader.load()`

**框架层的责任**：提供 `configModule`，把 10 个 Loader 的产物登记为 Koin `single`，范例直接 `get()` 即可。

**交付物**：新增 `data/ConfigModules.kt`（与 `DatabaseModules.kt` 同级），导出 `configModule`。

**关键实现细节**：
1. **`CombatConfigLoader` 返回 `CombatConfigResult`**（含 `combatConfig: CombatConfig` 和 `enemies: Map<String, EnemyDefinition>` 两部分）。注册时要拆开：
   ```kotlin
   single<CombatConfigResult> { CombatConfigLoader.load() }
   single<CombatConfig> { get<CombatConfigResult>().combatConfig }
   single<Map<String, EnemyDefinition>>(ConfigQualifiers.EnemyDefinitions) {
       get<CombatConfigResult>().enemies
   }
   ```
2. **泛型类型擦除问题**：`AttributeConfigLoader.load()` 返回 `Map<String, AttributeDefinition>`，`enemies` 也是 `Map<String, X>`，在 Koin 里直接 `single<Map<...>>` 会冲突。Kotlin `typealias` 在运行时被擦除为原始类型**不能**用于区分，**必须**用 `named()` 限定符。框架统一封装为 `ConfigQualifiers` 常量对象（见实际实现 [data/ConfigModules.kt](../src/main/kotlin/org/textrpg/application/data/ConfigModules.kt)）。
3. **Map qualifier 最小化**：仅当 Map 类型有**真实下游消费方**时才追加 qualifier。当前框架内有 4 个真实消费需求：`AttributeDefinitions` / `BuffDefinitions` / `EnemyDefinitions`（CombatSessionFactory）+ `QuestDefinitions`（QuestManager）。其他配置通过 data class 包装类型（如 `MapConfig` / `NpcConfig`）消费即可，不必为每个 Map 都加 qualifier。
4. **配置来源路径**：所有 Loader 都有 `DEFAULT_PATH` 默认值，Koin 注册使用默认即可；若未来需要测试环境或多配置文件，再引入配置源选择器

### F2 · `game` 层 Manager 缺少 Koin 模块

**现状**：
- `game/` 下 12 个子模块（attribute/buff/combat/command/effect/equipment/inventory/map/npc/quest/skill/ai）包含大量 Manager/Service/Engine
- 全部未进 Koin。范例要用必须按正确顺序手动 `new` 并传递依赖
- 这是 simple 分支 `loop()` 里出现大量手动 `val xxx = XxxManager(...)` 的根因
- 框架层最终注册 **13 个**全局单例（不含 `BuffManager` / `AttributeContainer`，因每实体一份属于范例层）

**框架层的责任**：为每个子模块提供 Koin 单例注册，汇总为 `gameModule`。范例只需 `get()`。

**需要谨慎的点**：
1. **`CoroutineScope`（gameScope）的注入位置**——`CombatSessionFactory` 依赖它。应放 `appModule`（基础设施），用 `single { CoroutineScope(Dispatchers.Default + SupervisorJob()) }`
2. **`BuffManager` / `CooldownManager` 的作用域** —— 框架只提供 `single` 的"全局实例"注册；如果范例需要"每玩家一份"（如 simple 的设计倾向），由范例自行用 `factory` 重写或在 `PlayerManager` 中独立 new。框架不做业务决策。
3. **依赖关系必须是 DAG** —— 证据见附录 B

**交付物**：新增 `game/GameModules.kt`（位置：`game/` 顶层），导出 `gameModule`。

### F3 · `game/ai/` 缺少 Koin 模块（Step 7 真正落地点）

**现状**：
- `AISceneManager` / `AIToolRegistry` / `PlayerTagManager` / `BuiltinAITools` / `SafetyPromptProvider` / `FunctionCallingClient` / `OpenAIFunctionCallingClient` 共 7 个类齐全

**框架层的责任**：
1. 提供 `aiModule`，把 AI 基础设施注册为单例
2. 在模块初始化阶段调用 `BuiltinAITools.registerAll(toolRegistry, mapManager, npcManager, questManager)` 让内置工具自动就绪

- 注册时用 `AIToolRegistry().also { BuiltinAITools.registerAll(it, get(), get(), get()) }` 在 `init` 阶段完成工具注册

**阻塞依赖**：F1（`AIConfig`）和 F2（三个 Manager）必须先做完。

### F4 · 内置 Handler 类化约定缺失

**现状与设计判断**：
- `CommandHandler` 是 `fun interface`（`CommandHandler.kt:11`），SAM 允许 lambda/闭包，**这个设计本身是对的**
- `CommandHandlerRegistry.resolve()` 已经是 key-routed factory（`CommandHandler.kt:64-73`），支持 `builtin:` / `script:` 分派——**框架层的 factory 能力已完备**
- 真正缺的是：当一个 handler 有 5 个以上依赖时，闭包式写法让依赖散布在外层闭包里，难以 Koin 注入。框架**未提供"类化"的标准模式**

**框架层的责任**：
1. 提供可选接口 `NamedCommandHandler`（继承 `CommandHandler`，附带 `name` 属性），让类实现时自带注册名
2. 在 `CommandHandlerRegistry` 上提供批量注册能力（如 `registerAll(handlers: List<NamedCommandHandler>)`），对称于 `ScriptHandlerFactory` 的脚本路径
3. **不强制** —— 闭包仍然合法，简单 handler 继续走闭包

**接口设计细节**：
```kotlin
// NamedCommandHandler 必须是普通 interface，不能是 fun interface
// 因为含有 name 属性，SAM 转换不支持多成员
interface NamedCommandHandler : CommandHandler {
    val name: String
}
```
`CommandHandler` 保持 `fun interface`（SAM），闭包场景继续可用。`NamedCommandHandler` 只服务于"类化 + Koin 注入"场景。

**交付物**：
- 新增 `game/command/NamedCommandHandler.kt`（约 10 行接口）
- `CommandHandlerRegistry` 新增 `registerAll(handlers)` 方法
- 更新 KDoc，说明两种用法的适用场景

### F5 · Application 启动编排约定缺失

**现状**：
- develop 分支的 `Application.kt` 82 行，把"DI 启动 + 消息循环"揉在一起
- simple 分支把它撑到 1100+ 行。这不是巧合——框架没有给出"Application 层应该怎么拆"的骨架，范例就各凭感觉

**框架层的责任**：提供一个**最小化的**启动编排骨架，让范例有地方挂自己的播种逻辑/handler 注册/启动钩子，而不必污染 `Application.kt` 本身。

**设计原则**：
- `Application.kt`：只负责启动 Koin（注册模块）+ 委派给 Bootstrap
- `ApplicationBootstrap`（新增类）：提供可覆盖的钩子
  - `onBeforeStart()`：给范例做数据播种（simple 的 `ItemSeeder` 挂这里）
  - `registerHandlers()`：默认从 Koin 拉 `List<NamedCommandHandler>` 全部注册；范例可追加
  - `startLoop()`：启动 OneBot/CLI 消息循环
- 框架提供默认实现，范例继承或注入

**交付物**：新增 `application/ApplicationBootstrap.kt`，以及对应的 Koin 注册。

**注意**：不引入新的容器/生命周期框架，只是一个规范的启动编排类。

## 三、目标架构（框架视角）

```
┌────────────────────────────────────────────────────┐
│ Application.kt            启动 Koin → 交给 Bootstrap │
└────────────────────────────────────────────────────┘
                        ↓
┌────────────────────────────────────────────────────┐
│ ApplicationBootstrap                                │
│  onBeforeStart() → registerHandlers() → startLoop() │
└────────────────────────────────────────────────────┘
                        ↓
┌──────────────────────── Koin Modules ──────────────┐
│ appModule       基础设施（HttpClient/LLMClient/…）    │
│ configModule    ★新增：10 个 ConfigLoader             │
│ databaseModule  已有                                 │
│ repositoryModule 已有                                │
│ gameModule      ★新增：game/ 下 13 个 Manager/Engine  │
│ aiModule        ★新增：AI Agent 基础设施              │
│ handlerModule   由范例层按本约定实现（不在框架层交付）  │
└────────────────────────────────────────────────────┘
```

**框架层交付 6 个 Koin 模块**（`handlerModule` 归范例层），依赖关系：
```
appModule ─→ databaseModule ─→ repositoryModule
    │                             ↑
    └→ configModule ──┬───→ gameModule ─→ aiModule
                      │                     (aiModule 依赖 gameModule 因
                      │                     BuiltinAITools 需要 3 个 Manager)
                      └──────────────────→ aiModule
```

模块间依赖：`repositoryModule` 依赖 `databaseModule`；`gameModule` 依赖 `configModule` + `appModule`（CoroutineScope）+ `repositoryModule`（给 `InventoryService` / `EquipmentService`）；`aiModule` 依赖 `gameModule`（3 个 Manager）+ `configModule`（AIConfig）+ `appModule`（LLMClient）。

```

                                  ↑
configModule → gameModule → aiModule
                 ↑ (CoroutineScope 来自 appModule)
```

`CombatSessionFactory` 聚合依赖（SessionManager, SkillEngine, EffectEngine, EnemyAI, CoroutineScope, buffDefinitions, attributeDefinitions, enemies）也是 DAG。

### R2 · 范例层与框架层的"可覆盖点"边界

框架层提供的 `gameModule` 是"默认实现"。范例（如 simple）可能有特殊需求（BuffManager / CooldownManager 的 per-player 分配）。框架的做法：
- 所有注册用 `single`，**允许**范例在自己的模块中 `single<BuffManager>(override = true) { ... }` 覆盖
- 不在框架层内置 `factory` 范式，避免误导

### R3 · simple 分支的同步成本

本框架层改动完成后，simple 分支合并/变基 develop 时，必须把原来 `loop()` 里手动 `new` 的 14 行改为 `get()`。这是 [范例层待修改.md](范例层待修改.md) 的 Phase E1 工作。

### R4 · `fun interface` 保留的代价

保留 `CommandHandler` 为 `fun interface`，好处是脚本通道可以用 lambda 直接返回；代价是**类实现方式多了一层选择**。通过 KDoc 清晰指导即可化解，不需要改接口性质。

### R5 · Koin override 语义

Koin 的 `override = true` 需要显式声明，新手容易踩坑。覆盖统一示例放在 [项目组织规范.md §7.3](项目组织规范.md)，框架层各模块 KDoc 不再重复展开示例，仅指向规范。

典型场景：范例层覆盖 `CooldownManager`（"每玩家一份"）或 `ApplicationBootstrap`（追加 `ItemSeeder` 播种）。具体语法：
```kotlin
val sampleOverrideModule = module {
    single<CooldownManager>(override = true) { /* 范例自定义 */ }
}
startKoin { modules(appModule, gameModule, ..., sampleOverrideModule) }
```
