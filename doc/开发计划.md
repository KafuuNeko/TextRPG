# TextRPG 开发计划

本文档记录各阶段的详细设计方案，作为跨对话开发的核心参考。

## 设计总纲

本项目是一个**高度抽象、配置驱动**的文字 RPG 框架。核心理念：

- **底层引擎不硬编码任何具体游戏概念**（没有"HP"、没有"力量"、没有"火球术"）
- 所有游戏内容通过 **YAML 配置 + Kotlin 脚本 (.kts)** 定义
- 引擎只提供**抽象机制**（属性槽、效果管道、触发器、会话状态机……）
- 在独立分支上建立一个**通用 RPG 范例**，验证框架设计并作为参考实现

**双轨开发**：框架层（main 分支）和范例层（独立分支）并行推进。

**通用设计模式**：框架中几乎所有系统都遵循相同的分层思路：
- **引擎层**：提供内置原子能力（最常用的基础操作）
- **配置层**：设计者通过 YAML 组合原子能力，定义游戏内容
- **脚本层**：高级开发者通过 .kts 脚本覆盖/扩展任意逻辑

## Step 1：抽象属性引擎 + 指令路由

> 状态：设计完成，待实现

### 1.1 抽象属性引擎

#### 核心概念

引擎将所有数值统一为**"属性"（Attribute）**，不区分资源 / 属性 / 状态。每个属性就是一个 key-value 对，通过配置赋予语义。

**全自由槽设计**：引擎不预设任何功能性分类（没有硬编码的"Life_Resource"概念）。Life / Action / Growth 只是配置层的约定，底层不做特殊处理。所有触发逻辑（归零死亡、耗尽拦截等）通过触发器配置实现。

属性分为两级：

| 级别 | 说明 | 举例 |
|------|------|------|
| **一级属性（Primary）** | 纯存储值，不由公式计算，只能直接赋值或通过修正器修改 | 力量、敏捷、体质 |
| **二级属性（Derived）** | 由公式从其他属性计算而来，支持嵌套引用其他二级属性 | 攻击力、暴击率、生命值上限 |

#### 修正器栈（Modifier Stack）

每个属性维护一个**分层修正器栈**：

```
最终值 = compute( 基础值 + Σ修正器 )
```

修正器结构：
- `source`: 来源标识（装备ID / BuffID / 脚本标识）
- `type`: 加算（flat）/ 乘算（percent）
- `value`: 修正值
- `priority`: 计算优先级（先加后乘 or 自定义顺序）

装备穿上 → 挂载修正器；脱下 → 按 source 移除。Buff 同理。引擎自动汇算最终值，无需脚本手动加减。

#### 公式系统

二级属性通过公式计算，公式定义在 YAML 中：

```yaml
formula: "strength * 2 + equipment_bonus"
```

- 引擎内置默认公式解析器（支持四则运算、属性引用）
- 脚本可覆盖公式逻辑（优先级高于配置公式）
- 二级属性可引用其他二级属性（引擎处理计算顺序 / 检测循环依赖）
- **双轨设计**：普通游戏作者用配置公式，高级开发者写脚本

#### 触发器系统

属性变化时可触发事件，引擎内置常见触发器：

| 触发器 | 说明 |
|--------|------|
| `on_deplete` | 当前值降至 0（或配置的下限）时触发 |
| `on_threshold` | 当前值达到指定阈值时触发 |
| `on_cap` | 当前值达到上限时触发 |
| `on_change` | 每次值变化时触发 |

触发器可指向：内置行为（如 `death`、`block_commands`）或自定义脚本。

#### YAML 属性定义示例

```yaml
attributes:
  # --- 一级属性 ---
  strength:
    tier: 1
    display_name: "力量"
    description: "物理攻击的基础"
    default: 10
    min: 0
    max: 9999

  agility:
    tier: 1
    display_name: "敏捷"
    default: 10
    min: 0
    max: 9999

  # --- 二级属性 ---
  max_hp:
    tier: 2
    display_name: "生命上限"
    formula: "constitution * 10 + 50"
    triggers:
      on_change: "recalc_current_hp_cap"

  current_hp:
    tier: 2
    display_name: "生命值"
    formula: null                           # 二级属性也可以不用公式，直接存值
    bound_max: "max_hp"                     # 当前值不超过 max_hp
    triggers:
      on_deplete:
        action: "death"
      on_threshold:
        value: "max_hp * 0.2"
        action: "script:low_hp_warning.kts"

  physical_attack:
    tier: 2
    display_name: "攻击力"
    formula: "strength * 2 + weapon_bonus"

  crit_rate:
    tier: 2
    display_name: "暴击率"
    formula: "agility * 0.5 + physical_attack * 0.01"   # 嵌套引用二级属性
    min: 0
    max: 100

  # --- 资源型属性 ---
  action_point:
    tier: 1
    display_name: "行动力"
    default: 100
    min: 0
    max: 100
    triggers:
      on_deplete:
        action: "block_commands"
    regen:
      amount: 10
      interval: "5m"

  exp:
    tier: 1
    display_name: "经验值"
    default: 0
    triggers:
      on_threshold:
        value: "level_up_threshold"
        action: "script:level_up.kts"
```

#### 数据存储

- **内存**：`Map<String, AttributeInstance>`，每个 AttributeInstance 含 baseValue + modifierStack + cachedFinalValue
- **数据库**：Player 表的 JSON 字段，落盘时序列化
- **静态配置**：YAML 文件定义属性模板

### 1.2 指令路由系统

#### 核心设计

指令系统完全**配置驱动**，设计者通过 YAML 注册指令。

#### 指令格式

- **前缀可配置**：默认 `/`，可改为 `!`、`#` 或任意字符串
- **格式**：`{前缀}{指令名} [参数1] [参数2] ...`
- **别名支持**：一个指令可有多个触发词

#### 指令 YAML 定义示例

```yaml
command_prefix: "/"

commands:
  status:
    display_name: "查看状态"
    aliases: ["状态", "st", "info"]
    description: "查看角色当前属性"
    handler: "builtin:status"
    requires:
      registered: true

  move:
    display_name: "移动"
    aliases: ["移动", "go", "走", "往"]
    params:
      - name: "target"
        type: "string"
        required: true
    handler: "builtin:move"
    requires:
      registered: true
      min_attribute:
        action_point: 5
    cost:
      action_point: 5

  custom_gather:
    display_name: "采集"
    aliases: ["采集", "采"]
    handler: "script:gather_handler.kts"
    requires:
      registered: true
      at_node_tag: "gatherable"
    cost:
      action_point: 15
```

#### Requires 统一机制

指令路由和技能共用同一套 Requires 检查器。内置常见检查：

| 检查项 | 说明 |
|--------|------|
| `registered` | 是否已注册角色 |
| `min_attribute` | 属性最低值检查（如行动力 >= 5） |
| `in_session` | 是否处于指定会话中（如战斗中） |
| `not_in_session` | 是否不在会话中 |
| `at_node_tag` | 当前地图节点是否有指定标签 |
| `has_item` | 是否拥有指定物品 |
| `has_buff` / `not_has_buff` | Buff 状态检查 |

可通过脚本扩展自定义检查条件。

#### 指令处理流程

```
玩家消息 → 前缀匹配 → 指令解析（名称 + 参数）
  → 查找指令定义（名称 / 别名匹配）
    → [未找到] → 如在会话中，转交会话处理；否则忽略或提示
    → [找到] → Requires 校验
      → [不满足] → 返回提示文本
      → [满足] → 扣除 cost → 路由到 handler
        → builtin:xxx → 内置处理器
        → script:xxx.kts → 脚本处理器
```

#### 会话模式（Session）

日常交互用独立指令；特定场景（战斗、商店、对话）进入**会话模式**：

- 玩家进入会话后，后续消息被该会话截获
- 会话内部有自己的指令集（如战斗中的"技能/道具/逃跑"）
- 会话通过协程 `suspend` 实现等待，不阻塞其他玩家
- 会话超时自动退出
- 设计者可通过脚本定义自定义会话类型

## Step 2：特效引擎（Effect Engine）

> 状态：设计完成，待实现

### 2.1 核心概念

特效引擎是框架的**"动词系统"**——属性引擎管"名词"（实体有什么），特效引擎管"动词"（对实体做什么）。

所有游戏行为（技能、消耗品、Buff 周期效果、地图事件……）最终都归结为一组 Effect 的执行。

### 2.2 原子操作（Atomic Effects）

引擎内置一批**原子操作**——最底层的"动词"，不可再分解：

| 原子操作 | 说明 | 参数示例 |
|----------|------|----------|
| `modify_attribute` | 修改目标属性值（加减） | target, attribute, amount |
| `set_attribute` | 直接设置属性值 | target, attribute, value |
| `add_modifier` | 挂载属性修正器 | target, attribute, source, type, value |
| `remove_modifier` | 移除属性修正器 | target, attribute, source |
| `add_buff` | 给目标添加 Buff | target, buff_id, stacks, duration |
| `remove_buff` | 移除目标的 Buff | target, buff_id |
| `teleport` | 传送到指定地图节点 | target, node_id |
| `send_message` | 向目标发送文本消息 | target, message |
| `give_item` | 给予物品 | target, item_template_id, quantity |
| `remove_item` | 移除物品 | target, item_template_id, quantity |
| `start_session` | 启动会话（如进入战斗） | target, session_type, params |
| `run_script` | 执行自定义脚本 | script_path, context |

设计者通过**组合原子操作**拼出复杂效果；脚本可注册自定义原子操作。

### 2.3 目标选择器（Target Selector）

每个 Effect 通过 `target` 字段指定作用对象。引擎内置选择器：

| 选择器 | 说明 |
|--------|------|
| `self` | 施法者自身 |
| `target` | 当前选中的目标 |
| `all_enemies` | 所有敌方 |
| `all_allies` | 所有友方 |
| `random_enemy` | 随机一个敌方 |
| `random_ally` | 随机一个友方 |
| `all` | 场上所有实体 |

脚本可注册自定义选择器（如"HP 最低的友方"）。

### 2.4 技能引擎

框架定义**技能（Skill）**的抽象结构：

```
技能 = Requires（前置条件） + Cost（消耗） + Effects（效果列表） + Cooldown（冷却）
```

复用 Step 1 的统一 Requires 机制。

#### 技能 YAML 定义示例

```yaml
skills:
  fireball:
    display_name: "火球术"
    description: "向敌人投掷一个火球"
    requires:
      not_has_buff: "silenced"
    cost:
      mp: 25
    cooldown: 2                    # 冷却 2 回合
    effects:
      - type: "modify_attribute"
        target: "target"
        attribute: "current_hp"
        amount: "-(physical_attack * 1.5 + 30)"   # 公式支持
      - type: "add_buff"
        target: "target"
        buff_id: "burn_dot"
        duration: 3
    custom_script: "scripts/fireball_special.kts"  # 可选：脚本后门

  heal:
    display_name: "治疗术"
    cost:
      mp: 20
    effects:
      - type: "modify_attribute"
        target: "self"
        attribute: "current_hp"
        amount: "intelligence * 2 + 50"

  potion_use:                      # 消耗品也是"技能"
    display_name: "使用药水"
    requires: {}
    cost: {}                       # 无 MP 消耗
    effects:
      - type: "modify_attribute"
        target: "self"
        attribute: "current_hp"
        amount: 50
    # 使用后扣减物品数量由背包服务处理
```

### 2.5 Effect 执行管道

```
触发源（指令/技能/Buff tick/地图事件/脚本）
  → Requires 校验
  → Cost 扣除
  → 遍历 Effects 列表
    → 解析 target（目标选择器）
    → 解析 amount/value（公式求值）
    → 执行原子操作
      → [成功] → 继续下一个 Effect
      → [失败] → 跳过，继续执行剩余 Effects
  → 执行 custom_script（如果有）
  → 记录 Cooldown
```

**失败策略**：单个 Effect 失败时跳过并继续，不回滚已执行的效果。

## Step 3：Buff / 状态异常系统

> 状态：设计完成，待实现

### 3.1 核心概念

Buff 本质上是**"挂在实体身上的、会随时间/回合触发的 Effect 集合"**。它是特效引擎的延伸。

Buff 也是**配置驱动**的，定义在 YAML 中。

### 3.2 Buff 数据模型

```yaml
buffs:
  burn_dot:
    display_name: "灼烧"
    description: "每回合受到火焰伤害"
    max_stacks: 5
    duration: 3                         # 持续 3 回合
    stack_policy: "stack"               # refresh / stack / independent
    # 挂载时的属性修正器（持续期间生效，移除时自动回滚）
    modifiers:
      - attribute: "fire_resist"
        type: "flat"
        value: -10                      # 降低火抗
    # 每回合触发的周期效果
    tick_effects:
      - type: "modify_attribute"
        target: "self"
        attribute: "current_hp"
        amount: "-10 * stacks"          # 伤害随层数增加
    # 生命周期钩子
    on_apply: "script:burn_apply.kts"   # 可选
    on_remove: null
    on_stack: null                      # 叠加时触发

  shield:
    display_name: "护盾"
    max_stacks: 1
    duration: 5
    stack_policy: "refresh"             # 重复施加时刷新持续时间
    modifiers:
      - attribute: "defense"
        type: "flat"
        value: 20
    tick_effects: []                    # 无周期效果
```

### 3.3 叠加策略

引擎内置三种，脚本可扩展：

| 策略 | 行为 |
|------|------|
| **Refresh** | 重复施加时刷新持续时间（取较长者），不增加层数 |
| **Stack** | 层数递增（直到 max_stacks），每层可能增强效果。持续时间可选刷新 |
| **Independent** | 每次施加独立存在，各自计时。允许同名 Buff 多个共存 |

### 3.4 Buff 生命周期

```
施加 Buff
  → 检查叠加策略
    → [Refresh] 刷新持续时间
    → [Stack] 增加层数，挂载额外修正器
    → [Independent] 新建独立实例
  → 挂载 modifiers 到目标属性修正器栈
  → 触发 on_apply

每回合结束时（战斗中）/ 每 tick（非战斗）
  → 执行 tick_effects
  → duration -= 1
  → duration == 0 → 移除 Buff

移除 Buff
  → 从修正器栈移除对应 modifiers（自动回滚属性）
  → 触发 on_remove
```

### 3.5 时间基准

Buff 持续时间基于**回合制**——每回合结束时 tick。

## Step 4：背包服务 + 装备逻辑

> 状态：设计完成，待实现

### 4.1 现有基础

数据层已完成：三层物品架构（Template → Instance → InventoryItem）、表结构、Repository。

本阶段补全**业务服务层**。

### 4.2 背包服务

#### 容量模式

框架支持可配置的容量限制模式：

```yaml
inventory:
  capacity_mode: "slots"     # slots（格子数）/ weight（负重）/ unlimited（无限）
  max_slots: 30              # 格子数模式下的上限
  max_weight: 100.0          # 负重模式下的上限
```

#### 核心操作

| 操作 | 说明 |
|------|------|
| `add_item` | 拾取/获取物品。可堆叠物品自动合并，不可堆叠创建新条目。检查容量 |
| `remove_item` | 丢弃/消耗物品。扣减数量，归零时删除条目 |
| `use_item` | 使用消耗品。触发物品关联的 Effect 列表（走特效引擎），然后扣减数量 |
| `move_item` | 背包内移动/整理 |
| `split_stack` | 拆分堆叠 |
| `transfer` | 跨容器转移（背包 → 仓库 / 邮件） |

### 4.3 装备服务

#### 穿脱逻辑

```
穿上装备
  → 检查装备槽位是否匹配
  → 检查等级要求（装备的 levelReq vs 角色等级属性）
  → 如果槽位已有装备 → 先脱下旧装备回背包
  → 从背包移除 → 放入装备位
  → 读取装备的属性加成 → 自动挂载为修正器（source = "equipment:{slot}"）

脱下装备
  → 检查背包是否有空间
  → 从装备位移除 → 放入背包
  → 按 source 移除对应修正器（属性自动回滚）
```

#### 装备属性加成

装备的属性加成定义在 ItemTemplate 或 ItemInstance 中：

```yaml
item_templates:
  iron_sword:
    type: "equipment"
    sub_type: "weapon"
    base_stats: '{"physical_attack": 15, "strength": 3}'
    # base_stats 中的 key 对应属性系统的 attribute key
```

穿上时，引擎遍历 base_stats，对每个 key 挂载 flat 类型修正器。

## Step 5：战斗系统

> 状态：设计完成，待实现

### 5.1 战斗模式

**纯回合制**：玩家和敌人交替行动，每回合选择一个动作。

### 5.2 战斗会话

战斗是一个**会话（Session）**——进入战斗后，玩家的消息被战斗会话截获处理。

```
触发战斗（遇敌/指令/事件）
  → 创建战斗会话
  → 初始化战斗上下文（参战实体、回合数、状态）
  → 进入战斗循环（协程 suspend）
```

### 5.3 回合流程

```
回合开始
  → 回合开始事件（触发"回合开始时"Buff 效果）
  → 判断行动方
    → [玩家回合]
      → 生成动作菜单（攻击/技能/道具/逃跑）
      → 发送菜单给玩家
      → suspend 等待玩家输入（超时自动防御）
      → 解析输入 → Requires 校验 → 执行 Effect 管道
    → [敌人回合]
      → AI 决策（规则配置 / 脚本）
      → 执行 Effect 管道
  → 回合结束事件（Buff tick、状态结算）
  → 检查胜负条件
    → [战斗结束] → 结算（经验/掉落/任务进度）→ 退出会话
    → [继续] → 下一回合
```

### 5.4 战斗结算公式

**配置 + 脚本**：框架提供默认的公式配置机制，脚本可覆盖。

```yaml
combat:
  # 默认伤害公式（可被脚本覆盖）
  damage_formula: "attacker.physical_attack - defender.defense * 0.5"
  # 暴击判定
  crit_check: "random(100) < attacker.crit_rate"
  crit_multiplier: 1.5
  # 最小伤害
  min_damage: 1
```

### 5.5 敌人 AI

**规则 + 脚本混合**：

```yaml
enemies:
  goblin:
    display_name: "哥布林"
    attributes:
      current_hp: 50
      max_hp: 50
      physical_attack: 8
      defense: 3
    skills: ["slash", "defend"]
    ai_rules:
      - condition: "self.current_hp < self.max_hp * 0.3"
        action: "defend"
        priority: 10
      - condition: "true"           # 默认行为
        action: "slash"
        priority: 1
    # 或者用脚本完全接管
    # ai_script: "scripts/goblin_ai.kts"

  dragon_boss:
    display_name: "远古巨龙"
    ai_script: "scripts/dragon_boss_ai.kts"   # 复杂 Boss 用脚本
```

### 5.6 战斗结算

```yaml
combat_rewards:
  exp_formula: "enemy.level * 10 + enemy.max_hp * 0.5"
  drops:
    # 掉落表也是配置驱动的
    - item: "goblin_ear"
      chance: 0.5
      quantity: "1-3"
```

## Step 6：NPC + 地图 + 任务

> 状态：设计完成，待实现

### 6.1 地图系统（有向图）

按照原始需求文档设计，采用**有向图（Directed Graph）**结构。

#### 节点数据结构

```yaml
# 地图节点定义（可 YAML 预设，也可由 AI 动态生成）
nodes:
  south_gate:
    display_name: "南门广场"
    description: "城市的南大门，行人络绎不绝。"
    tags: ["safe_zone", "shop"]
    connections:
      - target: "market_street"
        display: "→ 前往集市大街"
      - target: "south_road"
        display: "→ 出城南行"
        requires:
          min_attribute:
            level: 3
    entities:                               # 当前节点上的实体
      npcs: ["guard_wang", "merchant_li"]
      enemies: []                           # 安全区无敌人
    # 边界节点标记：玩家可在此触发 AI 动态生成新区域
    is_boundary: false
```

#### 移动逻辑

```
玩家发送 /移动 目标
  → 检查目标是否在当前节点的 connections 中
  → 检查连接的 requires（等级/物品/任务前置等）
  → 扣除 cost（行动力）
  → 更新玩家位置
  → 发送新节点的描述 + 可用动作
  → 如果新节点 is_boundary → 提示可以探索未知区域
```

#### AI 动态节点生成

```
玩家在边界节点发送 /探索
  → 构建上下文（当前节点信息、玩家等级、世界观设定）
  → 调用 LLM 生成新节点数据
  → 数据校验管道（格式/数值/合法性）
  → 写入数据库
  → 建立双向连接
  → 返回新区域描述
```

### 6.2 NPC 系统

框架层提供 NPC 实体定义 + 对话接口，AI 对话内容由 Step 7 支撑。

#### NPC 数据结构

```yaml
npcs:
  merchant_li:
    display_name: "商人老李"
    description: "一个精明的行商"
    location: "south_gate"
    prompt: "你是一个精明但友善的行商，在南门广场摆摊卖杂货。你喜欢讨价还价。"
    functions:                              # NPC 提供的功能
      - type: "shop"
        shop_id: "li_general_store"
      - type: "dialogue"                    # AI 驱动的自由对话
      - type: "quest_giver"
        quest_ids: ["fetch_herbs"]
    allowed_tools: []                       # AI 对话时可调用的工具（权限隔离）
```

#### 对话接口

```
玩家发送 @商人老李 你好啊
  → 定位 NPC（当前节点是否有该 NPC）
  → 进入 NPC 对话会话
  → 构建 AI prompt（NPC 的 prompt + 玩家 Tag 标签 + 对话历史摘要）
  → 调用 LLM 生成回复
  → 返回 NPC 回复文本
```

### 6.3 任务系统

**混合设计**：框架提供任务引擎（进度跟踪 + 目标检查 + 奖励发放），任务内容可以是 YAML 预设的，也可以由 AI 动态生成。

#### 任务数据结构

```yaml
quests:
  fetch_herbs:
    display_name: "采集草药"
    description: "商人老李需要 5 株野生草药"
```

## Step 7：AI Agent（LLM Function Calling）

> 状态：设计完成，待实现

### 7.1 AI 模块范围

框架层实现所有 AI 场景接口：

| 场景 | 说明 |
|------|------|
| **创世阶段** | GM 输入世界观 Prompt → AI 生成属性映射表 + 初始节点 + 初始 NPC |
| **角色创建** | 玩家自然语言描述 → AI 提取特征 → 分配初始属性 + 生成背景 |
| **动态节点生成** | 边界节点探索 → AI 生成新区域数据 |
| **NPC 对话** | 基于 NPC Prompt + 玩家 Tag → AI 生成对话文本 |
| **AI Boss 决策** | 战斗中 Boss 回合 → AI 选择技能（Function Calling） |

### 7.2 Function Calling 架构

```
AI 请求流程：
  构建上下文（快照数据 + 角色 Prompt + 可用 Tools）
    → 调用 LLM（Chat Completions + Function Calling）
    → 解析 AI 返回的 Tool Call
    → 权限检查（tool 是否在 allowed_tools 中）
    → 执行 Tool（走特效引擎 / 内置操作）
    → 返回结果给 AI（如需多轮）或输出给玩家
```

#### 工具定义

每个 AI 场景有独立的工具集：

```yaml
ai_tools:
  npc_dialogue:
    allowed_tools:
      - "send_message"                     # 只能说话
      # NPC 不能直接 give_item，必须走商店/任务系统
  
  boss_combat:
    allowed_tools:
      - "cast_skill"                       # 使用技能
      - "send_message"                     # 喊话
  
  world_generation:
    allowed_tools:
      - "create_node"                      # 创建地图节点
      - "create_npc"                       # 创建 NPC
      - "create_item_template"             # 创建物品模板
```

### 7.3 数据校验管道

**校验管道的接口在框架层，具体校验规则在范例层实现。**

框架提供校验管道的抽象接口：

```
AI 输出 → 格式校验（JSON/YAML 格式是否合法）→ 字段校验（必填字段是否齐全）→ 数值校验（是否在合法范围内）→ 权限校验（是否越权）→ 落库
```

框架内置：
- 格式校验（JSON/YAML 解析是否成功）
- 字段校验（根据目标数据结构的 schema 检查必填字段）
- 重试机制（校验失败时自动重新请求 AI）

范例层补充：
- 数值边界规则（具体的数值上下限取决于世界观设定）
- 权限隔离规则（具体哪些操作被允许取决于场景设计）

### 7.4 异步占位反馈

AI 请求耗时较长，框架内置占位反馈机制：

```
收到 AI 相关指令
  → 立即返回占位文本（"系统正在推演前方区域..."）
  → 协程异步执行 AI 请求
  → 完成后推送正式结果
```

### 7.5 预置安全限制（System Prompt Safety Layer）

框架在所有 AI 调用中**默认注入安全限制 System Prompt**，写在代码层面：

**默认限制内容（内置于框架代码中）：**
- 禁止生成违法、暴力、色情等不当内容
- 禁止 NPC AI 输出系统级操作指令（防越狱）
- 禁止 AI 生成超出 allowed_tools 范围的 Tool Call
- 限制 AI 输出长度，防止 Token 滥用
- 强制 AI 输出符合规定的结构化格式

**可移除，后果自负：**
- 设计者可以在配置中关闭或修改预置安全限制
- 框架代码中附带明确的**免责声明**注释
- 移除安全限制后产生的内容安全问题由设计者自行承担

```yaml
ai_safety:
  enabled: true                    # 默认启用，设计者可关闭
  preset_system_prompt: true       # 是否注入框架预置的安全 System Prompt
  custom_safety_rules: []          # 设计者可追加自定义安全规则
  # WARNING: 关闭 ai_safety 后，框架不对 AI 生成内容的安全性负责。
  # 详见 LICENSE 中的免责条款。
```

**代码层面**：安全 Prompt 硬编码在框架源码中，配置只是控制"是否启用"。即使设计者删除 YAML 配置，默认行为仍然是启用。要完全移除必须修改源码——这是有意为之的设计，确保安全限制不会被意外关闭。

### 7.6 上下文记忆压缩

不向 AI 传输完整聊天记录。框架维护玩家的**轻量级 Tag 系统**：

```yaml
# 玩家 Tag 示例（由系统自动分析或脚本生成）
```
