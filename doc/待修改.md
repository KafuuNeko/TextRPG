## 一、问题清单（按严重性排序）

### P0 · Step 7 AI Agent 子系统完全未接入

**现象**：进度表标注「Step 7 ✅ 完成」，但以下 6 个类**仅在自己文件里出现，主启动链从未实例化它们**：

| 类 | 现状 |
|---|---|
| `AISceneManager` | 定义了 5 种 AI 场景执行入口，零调用 |
| `PlayerTagManager` | 玩家 Tag 系统，零调用 |
| `AIToolRegistry` | 工具注册表，零调用 |
| `BuiltinAITools` | 5 个内置工具（`send_message`/`cast_skill`/`create_node`/`create_npc`/`create_quest`），零调用 |
| `SafetyPromptProvider` | 安全 Prompt 硬编码层，零调用 |
| `FunctionCallingClient` / `OpenAIFunctionCallingClient` | Function Calling 客户端，零调用 |

**后果**：
- `/对话` 在 `Application.kt` 里直接用 `llmClient.generate(...)`，**绕过**了安全 Prompt、玩家 Tag、工具白名单、多轮工具调用等整套基础设施
- `/探索` 同样绕过，直接裸 `generate`，手工解析 JSON
- Function Calling 能力根本没跑起来——Boss AI 决策、NPC 动态派发任务、AI 动态创建 NPC/物品模板等设计目标目前**全是空头支票**

**根因**：没有 DI 中心 → 新模块接入靠开发者"记得手动 new" → Step 7 这整块被漏掉，几个月内无人察觉。

---

### P1 · Application.kt 严重膨胀

**现状**：`Application.kt` **1100+ 行**，核心结构：

| 段落 | 行数 | 内容 |
|---|---|---|
| `loop()` | ~160 行 | 手动 new 14 个 manager + 启动消息循环 |
| `registerHandlers()` | ~800 行 | 11 个指令的闭包 handler 挤在一个函数里 |
| 私有辅助方法 | ~140 行 | `showEquipment` / `generateNpcDialogue` / `parseGeneratedNode` / `autoSelectSlot` / `parseSlotName` / `slotDisplayName` |

**问题**：
- 所有 handler 的业务逻辑**通过闭包捕获 14 个 manager 参数**——依赖关系散布在闭包里，人眼追踪困难
- 新增一个指令需要：改 commands.yaml + 在 `registerHandlers` 里加一大段闭包 + 可能改 `registerHandlers` 签名 + 传参链
- 指令逻辑**无法独立测试**（必须构造整个 loop() 环境）
- 阅读成本高——找"移动指令怎么实现的"要在 1100 行文件里定位

---

### P2 · Game 层 Manager 未进 DI

**现状**：

| 进 Koin 了 | 没进 Koin（全在 loop 里手动 new） |
|---|---|
| HttpClient、AppConfig、KotlinScriptRunner、OneBotAdapter、LLMClient | `SessionManager`、`CommandHandlerRegistry`、`CommandRouter` |
| DatabaseFactory、Database | `MapManager`、`NpcManager`、`QuestManager` |
| PlayerRepository、ItemRepository、BuffRepository | `InventoryService`、`EquipmentService` |
| | `EffectEngine`、`SkillEngine`、`CooldownManager`、`EnemyAI` |
| | `CombatSessionFactory`、`PlayerManager` |

**问题**：
- Manager 之间的依赖关系散布在 loop() 的顺序赋值语句里
- 新增 Manager 需要手工插入到 loop() 正确位置并接入 registerHandlers 签名
- 朋友指出的架构问题的直接证据

---

### P3 · CommandHandler 以闭包形式注册，不是独立类

**现状**：`registerHandlers` 里每个指令是一个 `CommandHandler { args, context -> ... }` 闭包 lambda，通过**捕获外层参数**获得依赖。

**问题**：
- 依赖不显式——要看 handler 用了哪些 manager 必须读实现
- 不能按需加载——所有 handler 必须在 `registerHandlers` 里全部注册一遍
- 不能单元测试——handler 没有独立的构造器可注入 mock
- 扩展性差——如果想让脚本/插件动态添加指令，没有清晰的注册路径

---

### P4 · 缺少 CommandHandler Factory 机制

**现状**：先生提到"不同 command 有不同的执行器，通过 factory 取得"。当前 `CommandHandlerRegistry` 本质上是一个 `Map<String, CommandHandler>`，注册在初始化时硬编码发生。没有：
- 按指令类型动态选择执行器的能力
- 分组/分层注册（比如按"基础指令" / "战斗指令" / "AI 指令"分组管理）的结构

---

### P5 · 辅助逻辑散落在 Application

**这些逻辑本应独立成类，但目前挤在 `Application` 里**：

| 逻辑 | 当前位置 | 应在位置 |
|---|---|---|
| `buildRichNodeInfo`（地图全景视图） | `registerHandlers` 内的闭包 lambda | 独立的 `NodeInfoFormatter` 服务 |
| `pickFromList`（编号/名字匹配） | 同上 | 独立工具，`game/command/` 或 `utils/` |
| `parseGeneratedNode` | `Application` 私有方法 | AI 服务内部（`ExplorationService` 或 `AISceneManager` 扩展） |
| `generateNpcDialogue` | `Application` 私有方法 | 应走 `AISceneManager.npcDialogue`（P0 解决后自然消除） |
| `showEquipment` / `autoSelectSlot` / `parseSlotName` / `slotDisplayName` | `Application` 私有方法 | `EquipmentDisplayService` 或装备 Handler 内 |

---

### P6 · CooldownManager 归属不清

**观察**：
- `loop()` 里 `val cooldownManager = CooldownManager() // 全局 CD（敌人用）` 传给 SkillEngine
- `PlayerManager` 内部为**每个玩家单独 new 一个 CooldownManager** 存入 `CachedPlayerData`
- 两者共存，但代码中**谁该用哪个**没有明确规则

**风险**：重构时容易混淆。需要在 P1 解决过程中明确语义：是"全局一份+玩家私有一份"的设计，还是"一个就够了"的冗余。

## 二、目标架构

### 2.1 分层原则

```
┌─────────────────────────────────────────┐
│  Application.kt（< 100 行）              │
│  只做：启动 Koin → 委派给 Bootstrap       │
└─────────────────────────────────────────┘
                 ↓
┌─────────────────────────────────────────┐
│  ApplicationBootstrap                    │
│  - ItemSeeder 播种                       │
│  - 从 Koin 收集所有 Handler 注册到 Registry│
│  - 启动消息循环（OneBot / 控制台）          │
└─────────────────────────────────────────┘
                 ↓
┌─────────────────────────────────────────┐
│  Koin Modules                            │
│  appModule · configModule · databaseModule │
│  repositoryModule · gameModule · aiModule │
│  handlerModule                           │
└─────────────────────────────────────────┘
```

### 2.2 Koin 模块划分

| Module | 管理的类型 |
|---|---|
| **appModule**（已存在，扩展） | HttpClient, AppConfig, KotlinScriptRunner, OneBotAdapter, LLMClient, **CoroutineScope**(gameScope) |
| **configModule**（新增） | AttributeConfig, CommandConfig, MapConfig, InventoryConfig, SkillConfig, BuffConfig, CombatConfigResult, NpcConfig, QuestConfig, AIConfig |
| **databaseModule**（已存在） | DatabaseFactory, Database |
| **repositoryModule**（已存在） | PlayerRepository, ItemRepository, BuffRepository |
| **gameModule**（新增） | 14 个游戏 Manager（SessionManager, CommandHandlerRegistry, CommandRouter, MapManager, NpcManager, QuestManager, InventoryService, EquipmentService, EffectEngine, SkillEngine, CooldownManager, EnemyAI, CombatSessionFactory, PlayerManager）+ 新抽出的 NodeInfoFormatter |
| **aiModule**（新增，收尾 Step 7） | FunctionCallingClient, AIToolRegistry, PlayerTagManager, AISceneManager（+ 初始化时调用 BuiltinAITools.registerAll） |
| **handlerModule**（新增） | 11 个 CommandHandler 类（每个指令一个） |

### 2.3 Handler 类化（对齐既有 Factory 模式）

**框架已有 Factory 基础**：`CommandHandler.kt:81` 已定义 `ScriptHandlerFactory` 接口，`CommandHandlerRegistry.resolve(handlerId)` 实际上就是一个 key-routed factory（根据 `builtin:` / `script:` 前缀路由到不同的工厂流程）。本次重构**延续并补齐**这个模式，不引入新的 Factory 概念。

每个指令独立为一个类，统一实现 `CommandHandler` 接口：

```kotlin
// game/command/handler/MoveCommandHandler.kt
class MoveCommandHandler(
    private val mapManager: MapManager,
    private val questManager: QuestManager,
    private val nodeInfoFormatter: NodeInfoFormatter,
    private val listResolver: CommandListResolver
) : NamedCommandHandler {
    override val name: String = "move"   // 对应 commands.yaml 里 handler: "builtin:move"

    override suspend fun execute(args: List<String>, context: CommandContext): String? {
        // 现 registerHandlers 中 /移动 闭包的逻辑整体搬过来
    }
}
```

**注册流程**：由 `ApplicationBootstrap` 从 Koin 批量拉取并注册到既有 `CommandHandlerRegistry`：

```kotlin
class ApplicationBootstrap(
    private val registry: CommandHandlerRegistry,
    private val handlers: List<NamedCommandHandler>,   // Koin 在 handlerModule 中组装成 List 注入
    private val scriptHandlerFactory: ScriptHandlerFactory   // 保留脚本路径
) {
    fun registerAllHandlers() {
        handlers.forEach { registry.registerBuiltin(it.name, it) }
        registry.scriptHandlerFactory = scriptHandlerFactory   // 对齐手册 "脚本热更新" 哲学
    }
}
```

其中 `NamedCommandHandler` 是一个小接口（放 `game/command/handler/NamedCommandHandler.kt`）：

```kotlin
interface NamedCommandHandler : CommandHandler {
    val name: String
}
```

### 2.4 与框架 Factory 模式的关系说明

| 层次 | 职责 | 现状 | 本次重构 |
|---|---|---|---|
| `CommandHandlerRegistry.resolve()` | key-routed factory（按 `builtin:` / `script:` 分派） | ✅ 已有 | 保持不变 |
| `ScriptHandlerFactory` | 脚本处理器创建（动态编译 .kts） | ✅ 已有接口 | 本次**不动**，保留给未来脚本系统 |
| 内置 Handler 创建 | 当前是闭包捕获，没有显式工厂 | ❌ 闭包式 | **改为类 + Koin 构造器注入** |

**结论**：先生要的"通过 factory 取得执行器"**已在框架层存在**（Registry.resolve）。当前问题只是**内置 handler 端没用 Factory 友好的方式实现**（都是闭包），重构后内置 handler 也走"Koin 单例 → Registry 注册 → Registry.resolve 取出"的清晰路径，和脚本 handler 对称。

### 2.5 新增/搬家组件的层次归属（对齐规范 §1.2 与 §3.2）

| 组件 | 新建位置 | 归属层 | 理由 |
|---|---|---|---|
| `NodeInfoFormatter` | `game/scene/NodeInfoFormatter.kt` | game 层 · 新子模块 `scene` | 跨 Map/Npc/Quest/Combat 的**场景展示服务**，不属于任何单一模块 |
| `CommandListResolver` | `game/command/CommandListResolver.kt` | game/command | 编号↔名字解析，是 command 层的通用工具 |
| `NamedCommandHandler`（接口） | `game/command/handler/NamedCommandHandler.kt` | game/command 新子目录 | 指令处理器扩展接口 |
| `MoveCommandHandler` 等 11 个 | `game/command/handler/*.kt` | game/command 新子目录 | 每个指令一个类 |
| `ApplicationBootstrap` | `application/ApplicationBootstrap.kt` | application 层 | 规范 §2.1 规定 application 层职责是"应用编排、入口配置" |
| `configModule` | `data/ConfigModules.kt` | data 层 | 配置加载属于 data 层（规范 §2.1） |
| `gameModule` | `game/GameModules.kt` | game 层 · 顶层 | 统一汇总 game 层所有 manager 的注册 |
| `aiModule` | `game/ai/AiModule.kt` | game/ai | 与 AI 模块同包 |
| `handlerModule` | `game/command/handler/HandlerModule.kt` | game/command/handler | 与 Handler 类同包 |

**新增的包**：
- `org.textrpg.application.game.scene` — 新增子模块，存放跨模块场景展示服务
- `org.textrpg.application.game.command.handler` — 新增子目录，存放指令处理器实现

**规范同步**：这些新增需要更新 [项目组织规范.md §2.1 和 §7.1](项目组织规范.md)。详见 §五 完成标准。

### 2.6 对手册"双轨开发"的明确保护

手册强调"90% 的常规游戏功能收敛于数据驱动组件，剩余的复杂机制通过原生 Kotlin 脚本"。本次重构**明确保护**以下机制：

- `ScriptHandlerFactory` 接口及其在 `CommandHandlerRegistry` 中的作用**零改动**
- `CustomScript` 在 Skill/Buff 配置中的后门**零改动**
- `KotlinScriptRunner` 工具类**零改动**
- `EffectEngine.registerHandler` 支持脚本动态注册的能力**零改动**

Handler 类化只针对**内置 handler**，不影响外部脚本扩展路径。这一点在 §5 完成标准里会作为"必须验证"项。

## 四、风险与权衡

### R1 · 循环依赖

**检查**：`CombatSessionFactory` 需要 `SessionManager, SkillEngine, EffectEngine, EnemyAI, CoroutineScope, buffDefinitions, attributeDefinitions, enemies`。`PlayerManager` 需要 `PlayerRepository, attributeDefinitions, buffDefinitions, SessionManager, MapManager, InventoryService`。**无循环**，是 DAG。

### R2 · CoroutineScope 注入

`gameScope` 需要在 `appModule` 用 `single { CoroutineScope(Dispatchers.Default + SupervisorJob()) }` 注册。`CombatSessionFactory` 通过构造器接收。

### R3 · 运行时参数不进 DI

`playerId`、`replier`、`args` 是**每条消息**的运行时值，走 `CommandContext`，不走 DI。

### R4 · YAML 配置对象改通过 DI 提供

Loader 对象（`AttributeConfigLoader`）保持 object，在 `configModule` 里写：
```kotlin
single<Map<String, AttributeDefinition>> { AttributeConfigLoader.load() }
single { CommandConfigLoader.load() }
// ...
```

### R5 · 回归测试风险

项目没有自动化测试。Phase 4 是最大的风险点——11 个 handler 全部重写。缓解策略：
- 每迁移一个 handler 就手动测一次，不积攒
- 保留现有 `registerHandlers` 的完整副本作为参考（暂时存在一个 `@Deprecated` 方法中，Phase 4 全部完成后删除）
- 准备一份标准回归 checklist（见下文）

### R6 · P6 CooldownManager 歧义

在 Phase 1 实施时必须澄清：
- **方案 A（推荐）**：保持"每玩家一份 CD"（在 PlayerManager 内），**删除** loop() 里那个"全局敌人 CD"——SkillEngine 应从 context 取玩家的 CooldownManager
- **方案 B**：明确文档化"全局 CD 用于敌人，玩家 CD 用于玩家"
- 这个决策小月无法独自做，需要先生或朋友确认。先按 B 保留现状，Phase 1 加 TODO，等后续澄清

## 九、规范文档自身的反馈（非本次重构范围）

复查过程中发现的规范文档问题，记录以便独立处理：

### 9.1 §10 与 §2.1 结构矛盾

[项目组织规范.md §2.1](项目组织规范.md) 描述 `game/` 下各子模块是**扁平结构**：

```
game/combat/
└── *.kt  # CombatSession / CombatEntity / EnemyAI
```

实际代码也是扁平结构。

但同一文档 §10.1 却列出**嵌套结构**：

```
game/combat/
├── model/              # 战斗相关枚举和值对象
├── entity/             # 战斗相关实体
├── service/            # 战斗服务接口
└── repository/         # 战斗数据仓储
```

§10.2~10.4 对 quest / npc / map 也给出了同样嵌套结构，但代码全部走扁平。

**建议**：§10 是早期规划遗留，与现实脱节。选一：
- 方案 A：删除 §10（因 §2.1 已覆盖实际结构）
- 方案 B：保留 §10，但明确声明为"**未来理想结构**"，并添加"当前代码暂未按此结构组织"的免责说明
- 方案 C：按 §10 重构所有 game 模块（工作量巨大，不推荐）

小月倾向方案 A——§2.1 已充分描述扁平结构，§10 只会误导后来者。

### 9.2 §7.1 Koin 模块表格仅 1 行

表格只列了 `data/DatabaseModules.kt` 一行，这实际上是规范**尚未覆盖 game 层的 Koin 模式**。本次重构会补齐这部分——应借此机会把 §7.1 扩展完整。

### 9.3 §7.2 "appModules 最后注册" 与代码不一致

规范 §7.2 说 "appModules 最后注册（汇总）"，但 `Application.kt:79` 实际写：

```kotlin
startKoin { modules(appModules, databaseModule, repositoryModule) }
```

appModules 在**第一个**位置。Koin 对模块注册顺序不敏感（只要没有 named qualifier 冲突），但规范和代码不一致会让后人困惑。

**建议**：规范 §7.2 修正为"顺序不影响解析，但按依赖层级从基础到上层排列以便阅读"。
