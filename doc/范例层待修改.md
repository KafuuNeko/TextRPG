## 一、simple 分支的工程问题（重判）

原 P0~P6 中归属范例层的问题：

| 原编号 | 问题 | 严重度 |
|---|---|---|
| P1 | `Application.kt` 1100+ 行：`loop()` 160 行、`registerHandlers()` 800 行、私有辅助 140 行 | 高 |
| P3（范例部分） | 11 个指令以闭包形式挤在 `registerHandlers()`，依赖靠闭包捕获 | 高 |
| P5 | `buildRichNodeInfo` / `pickFromList` / `parseGeneratedNode` / `generateNpcDialogue` / `showEquipment` / `autoSelectSlot` / `parseSlotName` / `slotDisplayName` 散落在 `Application` | 中 |
| P6 | `CooldownManager`：`loop()` 里一份"全局敌人 CD"、`PlayerManager` 内每玩家一份——语义不明 | 中（业务设计分歧） |
| AI 接入 | `/对话` 与 `/探索` 直接用 `llmClient.generate(...)` 裸调用，**绕过** `AISceneManager` 的 safety prompt / 工具白名单 / Tag context | 高（功能性而非架构性） |

**目标**：Application.kt 瘦身到 < 60 行（启动模块 + 调用 Bootstrap），11 个 handler 各自独立文件，AI 指令真正走 `AISceneManager`。

## 三、分阶段实施

### Phase E1 · 切换到框架层的 DI（接入成本最低）

**前置**：框架层 F1~F3 完成。

**改动**：
- `Application.kt` 的 `startKoin { modules(...) }` 加入 `configModule / gameModule / aiModule`

- `registerHandlers` 签名不动，**闭包内容不动**（保证行为一致，只换依赖来源）
- `ItemSeeder.seed(...)` 先留在 `Application.init` 或 `loop()` 起始处

**验证**：手动跑 `/注册 /移动 /战斗 /对话 /探索 /任务 /商店`，通过第五节 Checklist 的子集验证行为一致（本项目无自动化测试，"完全一致"以 Checklist 通过为准）。

**前置**：E1 完成。

**改动**：
- 新增 `game/scene/NodeInfoFormatter.kt`：
  - 把 `Application` 里的 `buildRichNodeInfo` 搬过来

### Phase E4 · AI 真正接入

**前置**：E1 完成（`AISceneManager` 已在 Koin）；E3 进行中或完成（`DialogueCommandHandler` / `ExploreCommandHandler` 已类化）。

**改动**：
- `DialogueCommandHandler`：
  - 把 `Application.generateNpcDialogue` 逻辑改为调用 `aiSceneManager.npcDialogue(playerId, npcPrompt, playerMessage)`

  - 保留"已有节点避重 + 敌人白名单 + JSON 解析"这些**业务规则**（属于 simple 特化需求，不是通用 AI 能力）

- 进度表更新：Step 7 从"接入就绪"→"✅ 真实运行"

**注意**：
- `/对话` 回复风格可能有变化（多了 safety prompt + Tag context）——这是**期望中的**，不是回归

### Phase E5 · CooldownManager 语义澄清

**前置**：E3 完成。

**问题回顾**：
- `loop()` 里 `val cooldownManager = CooldownManager() // 全局 CD（敌人用）` 传给 `SkillEngine`
- `PlayerManager` 内每玩家单独 `new` 一个 CooldownManager 存进 `CachedPlayerData`

**选择（需先生拍板）**：
- **方案 A**（推荐）：保持"每玩家一份 CD"——删除 `loop()` 里的"全局敌人 CD"，让 `SkillEngine` 从 context 取玩家的 CooldownManager；敌人用 `CombatSession` 内的临时 CD（或直接复用战斗会话生命周期）
- **方案 B**：明文化"全局 CD 用于敌人 + 玩家 CD 用于玩家"的双轨设计，写进 KDoc
- 本阶段实施前需先生或朋友确认

## 六、风险与权衡

### R1 · 行为兼容性

项目没有自动化测试。E3 是最大风险点——11 个 Handler 全部重写。

**缓解**：
- 分批迁移，每批 2~3 个
- 保留原 `registerHandlers()` 为 `@Deprecated` 副本，全部迁完再删
- 每批迁完立刻跑相关 Checklist 子集

### R2 · `/对话` 行为变化

E4 后 `/对话` 加入 safety prompt 和 Tag context，回复风格**会变**。这是**期望内变化**，不算回归——但需要先生体验后确认新风格是否可接受，否则调整 `ai.yaml` 里的 prompt。

### R3 · `/探索` 方案 A vs B

推荐方案 B（业务规则留在 Handler），避免污染框架层 `AISceneManager`。若 `AISceneManager.executeScene` 的通用签名无法满足 simple 对节点生成的定制需求，再考虑方案 A 并在框架层加通用钩子。

### R4 · Handler 依赖膨胀

`QuestCommandHandler` 依赖 5 个、`ShopCommandHandler` 依赖 4 个——还能接受。但若未来有 Handler 依赖 > 6 个，应进一步拆子服务（如"任务接取 / 任务交付"拆分为独立子命令或内部服务）。

### R5 · 与 develop 分支同步

本文档改动期间，develop 分支若有框架层新增（如新的 Manager），simple 需要做 `git merge develop` 或 `rebase develop`，并对应补 `handlerModule` 的依赖。建议：
- 每次 develop 有框架变更，24 小时内同步到 simple
- 避免 simple 分支长期脱离 develop 主线

## 七、不做什么

- **不动框架层**：框架代码任何改动都属于 [框架层待修改.md](框架层待修改.md)

- **不重写 AI Prompt**：`ai.yaml` 保持现状
- **不引入自动化测试**：手动 Checklist 足够，单独立项

### 附录 B · 重构后的 `Application.kt` 参考结构

```kotlin
object Application : KoinComponent {
    init {
        startKoin {
            modules(
                appModule, configModule, databaseModule, repositoryModule,
                gameModule, aiModule,
                handlerModule  // 范例侧
            )
        }
    }

    suspend fun run() {
        val bootstrap: ApplicationBootstrap = get()
        bootstrap.start()
    }
}

fun main() {
    runBlocking { Application.run() }
}
```

`handlerModule` 内部批量 bind 为 `NamedCommandHandler`；`ApplicationBootstrap` 在 `registerHandlers()` 钩子里调用 `getKoin().getAll<NamedCommandHandler>()` 拿到全部实例并批量注册到 `CommandHandlerRegistry`。
