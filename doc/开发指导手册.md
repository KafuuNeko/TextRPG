# TextRPG-AI 框架官方开发与架构手册 (现代 Kotlin 架构版)

## 核心设计哲学
**“协程构筑非阻塞高并发引擎，YAML/DSL 拼装游戏血肉，原生 Kotlin Scripting (.kts) 提供极速热更新，大语言模型 (LLM) 赋予世界自主心智。”**

本框架采用现代化的 Kotlin JVM 生态。90% 的常规游戏功能（战斗、数值、机制）收敛于数据驱动组件，剩余的复杂机制通过原生的 Kotlin 脚本完美填补。框架设计的最高准则为**“极致解耦与极简并发”**。

---

## 第一章：技术栈与基础设施标准

底层的技术选型全面拥抱 JetBrains 官方生态，确保极致的类型安全、异步性能与开发体验：

* **核心语言**：Kotlin (运行于 JVM，可享受庞大的 Java 生态支持)。
* **网络与并发核心**：
    * **协程 (`kotlinx.coroutines`)**：框架的并发灵魂。以万级并发、极低开销处理大模型请求与玩家回合等待，杜绝线程阻塞。
    * **Ktor 框架**：一站式搞定反向 WebSocket 监听（对接 OneBot）和异步 HTTP 请求（对接 LLM API）。
* **数据解析与配置**：
    * **`kotlinx.serialization`**：编译期安全的序列化库，自动将 YAML 或 JSON 映射为 Kotlin Data Class，彻底告别空指针异常。
* **动态扩展桥梁**：
    * **Kotlin Scripting (`.kts`)**：直接将一段 Kotlin 代码作为脚本在运行时动态加载编译。提供强大的类型检查与 IDE 代码补全支持。
* **数据持久化**：
    * **JetBrains Exposed**：轻量级 SQL DSL 框架。以写原生 Kotlin 代码的方式操作 SQLite 数据库，类型安全且极其优雅。

## 第二章：核心范式 —— 数据驱动的特效引擎 (Effect Engine)

通过 Kotlin 强大的数据类（Data Class）和密封类（Sealed Class），我们可以极其优雅地实现组件化引擎，将“动词”变为可配置的模块。

### 1. 结构化定义规范 (YAML)
游戏内的技能、消耗品统一采用 **“拦截器 (Requires) + 执行流水线 (Effects) + 后门扩展 (Custom Script)”** 的三段式结构：

```yaml
# 嗜血火球术配置示例
skills:
  fireball_leech:
    name: "嗜血火球"
    # 1. 拦截器：预校验玩家状态
    requires:
      minMp: 25
      notInState: ["silenced"]
    # 2. 效果流：按顺序执行组件
    effects:
      - type: "damage"
        element: "fire"
        baseValue: 50
      - type: "heal"
        target: "self"
        baseValue: 20
      - type: "addBuff"
        buffId: "burn_normal"
        duration: 3
    # 3. 脚本后门：运行时加载的特殊逻辑扩展
    customScript: "scripts/fireball_special.kts" 
```

### 2. Kotlin 层的强类型映射执行
利用 `kotlinx.serialization`，框架将上述 YAML 直接映射为对象体系：
```kotlin
@Serializable
data class Skill(
    val name: String,
    val requires: Requires,
    val effects: List<Effect>,
    val customScript: String? = null
)

@Serializable
sealed class Effect {
    @Serializable @SerialName("damage")
    data class Damage(val element: String, val baseValue: Int) : Effect()
    
    @Serializable @SerialName("heal")
    data class Heal(val target: String, val baseValue: Int) : Effect()
}
```
**执行管道**：引擎通过 `when(effect)` 模式匹配，依次路由各个组件逻辑，代码极度清晰，易于无限扩展新的 Effect 类型。

## 第三章：协程驱动的战斗状态机与多级菜单

在处理“等待玩家输入”这种跨回合、易卡死的操作时，Kotlin 协程展现出了无可比拟的优势。代码不仅如同步执行般直观，且完全不占用系统线程资源。

### 1. 挂起函数 (`suspend`) 的妙用
在玩家回合开始时，生成菜单并**挂起（Suspend）**当前协程，静默等待：

```kotlin
suspend fun processPlayerTurn(player: Player) {
    // 1. 根据当前状态和拥有技能，生成并下发合法菜单
    val menu = generateValidMenu(player)
    sendToPlayer(player.id, menu.renderText())

    // 2. 挂起当前协程，直到监听到该玩家的合法输入或触发超时防御
    val userInput = waitForPlayerInput(player.id, validOptions = menu.validKeys) 
    
    // 3. 协程恢复执行，路由到对应的特效引擎结算
    executeAction(player, menu.getAction(userInput))
}
```

### 2. 动态多级菜单路由
* 菜单生成器主动预检玩家的 MP 和状态，自动剔除（或置灰展示）当前不满足 `Requires` 条件的选项。
* 轻松实现“主菜单 -> 技能分类 -> 具体技能 -> 目标选择”的无缝下钻与返回体验。

## 第四章：大一统技能与状态异常系统

### 1. 万物皆技能 (Universal Skill)
普通攻击、魔法、吃药抽象为统一定义的 `Skill` 接口。底层统一经过 `校验判定 -> 触发 Effect 引擎 -> 扣除消耗 -> 结算 CD` 的标准管线。

### 2. 策略化 Buff 系统
利用 Kotlin 集合强大的操作符，优雅处理复杂的 Buff 叠加规则：
* **`Refresh` (刷新机制)**：`buffs.find { it.id == newBuff.id }?.let { it.duration = max(it.duration, newBuff.duration) }`
* **`Stack` (层数递增)**：同一 Buff 叠加层数，可配合脚本或组件实现“叠满 5 层引爆真实伤害”。
* **`Independent` (独立存在)**：允许存在多个相互独立的同名 Buff。
* **生命周期Tick**：每回合结束时或依托于全局时间轴协程，安全遍历所有实体结算周期性流血/恢复效果。

## 第五章：AI Agent 与 Function Calling 架构

结合 Ktor Client，让 NPC 真正具备不卡顿的智能与反向改变游戏世界的能力。

### 1. 异步决策流
当轮到 AI Boss 行动时，生成战况快照并请求大模型：
```kotlin
suspend fun processAiTurn(boss: Npc, player: Player) {
    val snapshot = buildBattleSnapshot(boss, player)
    
    // 发起 HTTP 请求，当前协程挂起，服务器主线程继续流畅处理其他上千个玩家的动作
    val aiResponse = llmClient.requestFunctionCalling(boss.prompt, snapshot)
    
    // 收到 AI 决策，协程恢复
    sendToPlayer(player.id, aiResponse.speech)
    
    // 二次鉴权并执行 AI 调用的 Tool (如 cast_fireball)
    if (boss.hasTool(aiResponse.toolName)) {
        effectEngine.process(aiResponse.toolName, aiResponse.args)
    }
}
```

### 2. 边界安全防线
* **工具白名单**：在 NPC 的配置中严格声明允许的 `allowed_tools`。
* **零信任原则**：把 AI 传回的参数当作玩家的恶意输入对待，调用 Effect 前必须强制穿透 `Requires` 拦截器校验。

## 第六章：数据持久化 (JetBrains Exposed)

使用 Exposed 提供的 DSL 管理玩家存档，彻底告别拼写原生 SQL 字符串。

```kotlin
// 数据库表结构定义
object Players : IntIdTable() {
    val qqId = varchar("qq_id", 50).uniqueIndex()
    val level = integer("level").default(1)
    val hp = integer("hp")
    val locationId = varchar("location_id", 100)
}

// 挂起函数执行异步存档写入
suspend fun savePlayerState(playerInfo: Player) = newSuspendedTransaction {
    Players.update({ Players.qqId eq playerInfo.qqId }) {
        it[level] = playerInfo.level
        it[hp] = playerInfo.hp
        it[locationId] = playerInfo.locationId
    }
}
```
**落地策略**：内存中维护高频战斗状态。仅在战斗结束、地图切换、掉落结算、下线等低频安全节点，调用挂起函数异步 Flush 到 SQLite，消除所有 I/O 瓶颈。

## 第七章：开发实施路线图 (Milestones)

按照组件独立、由静到动的原则稳步推进：

* **Phase 1: 基础设施 (Ktor & JSON)**
  搭建 Ktor 项目，连接 OneBot WebSocket，实现基础的指令路由与 `kotlinx.serialization` 序列化。
* **Phase 2: 核心组件引擎 (Effect Engine)**
  建立数据类模型，解析 YAML/JSON，实现 `Damage`, `Heal`, `AddBuff` 等核心组件的解析与执行。
* **Phase 3: 协程状态机 (Combat Flow)**
  利用 `suspend` 函数重构玩家输入等待逻辑，跑通动态多级菜单与回合制战斗循环。
* **Phase 4: 原生脚本注入 (Kotlin Scripting)**
  引入 `.kts` 动态编译模块，打通静态配置与外部运行时逻辑扩展的桥梁。
* **Phase 5: 数据落地 (Exposed ORM)**
  集成 SQLite 与 Exposed，实现核心数据的对象关系映射与异步存盘。
* **Phase 6: 唤醒智能体 (LLM Integration)**
  使用 Ktor HTTP Client 对接大模型。利用 Function Calling 让 AI 动态接管 Boss 回合和 NPC 任务派发，实现框架的最终形态。
