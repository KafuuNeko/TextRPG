## Step 1 实现详情

### 抽象属性引擎

| 子模块 | 文件路径 | 说明 |
|--------|----------|------|
| 属性枚举 | `domain/model/AttributeEnums.kt` | AttributeTier（PRIMARY/DERIVED）、ModifierType（FLAT/PERCENT）、TriggerType（4 种） |
| 属性模型 | `domain/model/AttributeModels.kt` | AttributeDefinition、TriggerDefinition、RegenDefinition、Modifier（全部不可变） |
| 配置加载 | `data/config/AttributeConfigLoader.kt` | YAML → 领域模型，通用 Map 解析，支持触发器简写/完整双格式 |
| 配置文件 | `resources/config/attributes.yaml` | 属性定义模板，含完整格式注释和示例 |
| 运行时实例 | `game/attribute/AttributeInstance.kt` | 单属性运行时：基础值 + 修正器栈 + 脏标记（dirty flag）缓存机制 |
| 属性容器 | `game/attribute/AttributeContainer.kt` | 实体属性管理：统一 API、boundMax 跨属性约束、二级属性拓扑排序自动重算、触发器分发 |
| 公式引擎 | `game/attribute/FormulaEngine.kt` | 递归下降解析器：四则运算、括号、一元负号、属性引用；依赖提取（用于拓扑排序） |
| 触发器 | `game/attribute/AttributeTrigger.kt` | TriggerEvent 事件数据 + TriggerHandler（fun interface） |

**核心设计点：**
- 引擎不硬编码任何游戏概念，所有语义通过 YAML 配置赋予
- 修正器计算：FLAT 累加 → PERCENT 加算合并后一次乘算 → 范围裁剪
- 二级属性通过 Kahn 拓扑排序保证依赖链的计算顺序
- 触发器采用跨越检测（crossing detection），防止重复触发
- boundMax 约束在 Container.getValue() 层统一处理，Instance 不感知其他属性

### 指令路由系统

| 子模块 | 文件路径 | 说明 |
|--------|----------|------|
| 指令模型 | `domain/model/CommandModels.kt` | CommandDefinition、ParamDefinition、RequiresDefinition |
| 配置加载 | `data/config/CommandConfigLoader.kt` | YAML 指令配置加载 + CommandConfig 数据类 |
| 执行上下文 | `game/command/CommandContext.kt` | 接口：游戏状态访问（属性、会话、物品、Buff、节点标签） |
| Requires 校验 | `game/command/RequiresChecker.kt` | 统一前置条件校验器，8 种内置检查 + RequireResult |
| 处理器 | `game/command/CommandHandler.kt` | CommandHandler 接口 + CommandHandlerRegistry + ScriptHandlerFactory |
| 主路由器 | `game/command/CommandRouter.kt` | 完整管道：会话截获 → 前缀匹配 → 解析 → 查找（O(1)索引）→ 校验 → 扣费 → 执行 |
| 会话框架 | `game/command/Session.kt` | Session 接口 + SessionManager（ConcurrentHashMap，线程安全） |
| 配置文件 | `resources/config/commands.yaml` | 指令定义模板，含完整格式注释和示例 |

**核心设计点：**
- 指令完全配置驱动，YAML 注册指令，引擎负责解析和路由
- Requires 机制与技能系统共用（统一的 RequiresDefinition）
- Cost 扣除保证原子性：先校验全部满足，再统一扣除
- Handler 支持 `builtin:xxx` 和 `script:xxx.kts` 双模式
- 会话模式优先级高于普通指令，进入会话后消息被截获
- 路由结果使用密封类（sealed class）表达所有终态

**核心设计点：**
- 运行时派发基于字符串匹配，BuiltinEffectType enum 仅作文档辅助
- `registerHandler` 支持脚本注册自定义效果类型
- EntityAccessor 与 CommandContext 分离：前者面向任意实体的原子操作，后者面向单玩家指令场景
- 尚未实现的系统（Buff/地图/战斗）用 stub 占位，后续 Step 接入后自动生效
- 单个 Effect 失败时跳过继续，不回滚已执行的效果

## Step 3 实现详情

### Buff 系统

| 子模块 | 文件路径 | 说明 |
|--------|----------|------|
| Buff 模型 | `domain/model/BuffModels.kt` | StackPolicy 枚举（3 种叠加策略）、BuffModifierDefinition、BuffDefinition |
| 配置解析 | `data/config/BuffConfigLoader.kt` | YAML 加载（BuffConfig + BuffConfigLoader），复用 EffectConfigLoader 和 RequiresParser |
| Buff 实例 | `game/buff/BuffInstance.kt` | 运行时状态：层数/持续时间/修正器来源标识 |
| Tick 上下文 | `game/buff/BuffTickEffectContext.kt` | Kotlin 委托包装 EffectContext，注入 stacks 公式变量 |
| Buff 管理器 | `game/buff/BuffManager.kt` | per-entity：施加/叠加策略/修正器挂载回滚/tick 周期效果/过期移除/脚本钩子 |
| 实体基类 | `game/buff/BuffAwareEntityAccessor.kt` | 抽象基类：属性委托 AttributeContainer + Buff 委托 BuffManager |

**核心设计点：**
- Buff = 持续性修正器 + 周期 Effect 集合，是特效引擎的延伸
- 三种叠加策略：Refresh（刷新时间）、Stack（层数叠加+修正器缩放）、Independent（独立实例）
- STACK 策略修正器按层数缩放：移除旧修正器 → 按 value * stacks 重新挂载
- BuffTickEffectContext 通过 Kotlin `by` 委托注入 stacks 变量，FormulaEngine 无需改动
- tick 由外部驱动（调用方负责触发），BuffManager 不自行定时
- BuffAwareEntityAccessor 通过组合模式将属性和 Buff 管理统一在 EntityAccessor 接口下
- 通过组合接入，未修改任何 Step 2 现有文件
